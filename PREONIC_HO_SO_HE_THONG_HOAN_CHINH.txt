========================================================================
1. TỔNG QUAN HỆ THỐNG
========================================================================

1.1. Mục tiêu nghiệp vụ
- PreOnic là nền tảng kết nối nông dân và doanh nghiệp trong bài toán bao tiêu nông sản.
- Hệ thống tập trung vào 6 nhóm nghiệp vụ:
  - Xác thực và phân quyền người dùng.
  - Quản lý sản phẩm nông sản.
  - Tạo, ký và theo dõi hợp đồng.
  - Escrow / ký quỹ / giải ngân theo milestone.
  - Ví ảo và nạp tiền qua SePay.
  - Nhắn tin, thông báo, cảnh báo thời tiết, đánh giá đối tác.

1.2. Kiến trúc tổng thể
- Backend:
  - Nền tảng: Express + TypeScript + Mongoose.
  - Điểm vào chính: `be/src/server.ts`, `be/src/app.ts`.
  - Có cron job cho thời tiết và nhắc doanh nghiệp xác nhận đơn hàng đã giao.
- Frontend:
  - Nền tảng: React 19 + `react-router-dom` 7 + Bootstrap + Framer Motion + Recharts.
  - Điểm vào chính: `fe/src/index.js`, `fe/src/App.js`.
  - Quản lý đăng nhập bằng `AuthContext` + `localStorage` + `axios interceptor`.
- Database:
  - MongoDB.
  - 12 model/chính sách dữ liệu chính: `User`, `Product`, `Review`, `Contract`, `Escrow`, `Dispute`, `PaymentTransaction`, `Notification`, `WeatherAlert`, `Conversation`, `Message`, `PartnerRating`.
- Lưu trữ file:
  - Ảnh/PDF upload lưu vào thư mục `be/uploads` và serve qua route `/uploads/*`.

1.3. Công nghệ và thư viện chính
- Backend:
  - `express`, `mongoose`, `jsonwebtoken`, `bcryptjs`, `cors`, `helmet`, `compression`, `cookie-parser`, `morgan`, `nodemailer`, `node-cron`.
- Frontend:
  - `react`, `react-router-dom`, `axios`, `bootstrap`, `react-bootstrap`, `framer-motion`, `recharts`.

1.4. Cấu hình runtime đang dùng trong workspace
- Backend:
  - `PORT=5001`
  - `API_PREFIX=/api/v1`
  - `FRONTEND_URL=http://localhost:3001`
- Frontend:
  - `REACT_APP_API_URL=http://localhost:5001/api/v1`
  - `PORT=3001`
- Tình trạng khớp:
  - `be/.env` và `fe/.env` hiện tại đang khớp nhau.
- Tình trạng lệch:
  - `be/.env.example` vẫn để `PORT=5000` và `FRONTEND_URL=http://localhost:3000`
  - `fe/.env.example` vẫn để `REACT_APP_API_URL=http://localhost:5000/api/v1`
  - `fe/package.json` vẫn giữ `proxy` là `http://localhost:5000`

- `npm run build` trong thư mục `be`: thành công.

1.7. Cơ chế phân quyền tổng quát

- Farmer:
  - CRUD sản phẩm của mình.
  - Xem dashboard farmer.
  - Tạo/ký hợp đồng ở vai trò nông dân.
  - Xác nhận milestone của nông dân trong escrow.

2.4. Milestone escrow theo `paymentTerms`

Template milestone có 5 bước
- Step 1: Ký quỹ - `requiredBy enterprise`
- Step 2: Chuẩn bị hàng hóa - `requiredBy farmer`
- Step 3: Giao hàng - `requiredBy farmer`
- Step 4: Kiểm tra chất lượng - `requiredBy enterprise`
- Step 5: Hoàn tất - `requiredBy system`

Tỷ lệ release hiện tại
- `paymentTerms=50_50` -> `[0, 0, 50, 50, 0]`
- `paymentTerms=30_70` -> `[0, 0, 30, 60, 10]`
- `paymentTerms=100_delivery` -> `[0, 0, 100, 0, 0]`
- `paymentTerms=100_upfront` -> `[100, 0, 0, 0, 0]`
- `fallback tùy chỉnh` -> `[0, 0, 40, 50, 10]`

2.5. Ngưỡng thời tiết mặc định
- `extremeHeatTemp`: 38 độ C
- `extremeColdTemp`: 5 độ C
- `heavyRainMm`: 100 mm/ngày
- `strongWindKmh`: 60 km/h
- `droughtMm`: 5 mm trong 14 ngày
- `droughtDays`: 14 ngày

3. THIẾT KẾ API
========================================================================

3.1. Quy ước chung
- Base URL thực tế hiện tại:
  - Backend local: `http://localhost:5001`
  - API prefix: `/api/v1`
  - Ví dụ đầy đủ: `http://localhost:5001/api/v1/products`
- Header bảo mật:
  - `Authorization: Bearer <accessToken>`
- Cookie:
  - `refreshToken` được lưu dạng `httpOnly cookie`.
- `Content-Type`:
  - `application/json`
- Định dạng success response phổ biến:
  - {
      "success": true,
      "status": "success",
      "message": "...",
      "data": ...
    }
- Định dạng error response phổ biến:
  - {
      "success": false,
      "status": "error",
      "message": "...",
      "error": "...",
      "stack": "..."
    }
- Hành vi khi DB mất kết nối:
  - Mọi route trừ `/health` trả `503` với message "Database is temporarily unavailable..."

3.3. Auth API

POST /api/v1/auth/register
- Quyền: Công khai
- Dữ liệu gửi lên:
  - `fullName`
  - `email`
  - `phone`
  - `province`
  - `district`
  - `ward`
  - `password`
  - `confirmPassword`
  - `role`
  - `agreeTerms`
- Validation:
  - email hợp lệ
  - phone 10-11 số
  - password tối thiểu 6 ký tự
  - role chỉ `farmer | enterprise`
- Phản hồi:
  - `data.user`
  - `data.accessToken`
  - `refreshToken` lưu cookie

POST /api/v1/auth/login
- Quyền: Công khai
- Dữ liệu gửi lên:
  - `emailOrPhone`
  - `password`
  - `rememberMe`
- Quy tắc:
  - Hỗ trợ đăng nhập bằng email hoặc số điện thoại.
  - Khóa tạm thời sau 5 lần sai.
- Phản hồi:
  - `data.user`
  - `data.accessToken`
  - `refreshToken` lưu cookie

4. HƯỚNG DẪN CÀI ĐẶT & CHẠY
========================================================================

4.1. Điều kiện tiên quyết
- Node.js: khuyến nghị 18+ hoặc 20+.
- npm.
- MongoDB local hoặc MongoDB Atlas.
- Nếu dùng SePay thật:
  - Cần có URL public cho backend để nhận webhook.
- Nếu dùng email thật:
  - Cần có `SMTP_HOST`, `SMTP_USER`, `SMTP_PASS`.

4.2. Cấu trúc thư mục
- `be`: backend API
- `fe`: frontend React
- `PROJECT_OVERVIEW.txt`: tài liệu cũ, không nên xem là source of truth

4.3. Cài đặt dependencies
- Backend:
  - `cd be`
  - `npm install`
- Frontend:
  - `cd fe`
  - `npm install`

6.1. Mẫu auth response kèm refresh token cookie
Nguồn: `be/src/controllers/auth.controller.ts`

```ts
const sendAuthResponse = (
  res: Response,
  statusCode: number,
  message: string,
  user: any,
  tokens: AuthTokens,
  cookieMaxAge: number = THIRTY_DAYS_MS
) => {
  res.cookie(REFRESH_TOKEN_COOKIE, tokens.refreshToken, getRefreshTokenCookieOptions(cookieMaxAge));
  res.status(statusCode).json({
    success: true,
    status: 'success',
    message,
    data: { user, accessToken: tokens.accessToken },
  });
};
```

Giải thích
- Access token được trả về trong JSON để frontend gắn vào `Authorization` header.
- Refresh token không trả thẳng ra `localStorage` mà được giữ trong `httpOnly cookie`.
- Cơ chế này giảm rủi ro lộ refresh token qua JavaScript phía client.

6.2. Axios interceptor tự động refresh access token
Nguồn: `fe/src/services/api.js`

```js
api.interceptors.response.use(
  (response) => response,
  async (error) => {
    const originalRequest = error.config;
    const isUnauthorized = error.response?.status === UNAUTHORIZED_STATUS;
    const hasNotRetried = !originalRequest?._retry;
    const isRefreshRequest = originalRequest?.url?.includes('/auth/refresh-token');

    if (isUnauthorized && hasNotRetried && !isRefreshRequest) {
      originalRequest._retry = true;

      try {
        const accessToken = await refreshAccessToken();
        originalRequest.headers = originalRequest.headers || {};
        originalRequest.headers.Authorization = `Bearer ${accessToken}`;
        return api(originalRequest);
      } catch {
        clearStoredAuth();
        redirectToLogin();
        return Promise.reject(error);
      }
    }

    return Promise.reject(error);
  }
);
```

Giải thích
- Khi access token hết hạn, frontend thử gọi `/auth/refresh-token`.
- Nếu refresh thành công, request cũ được gửi lại tự động.
- Nếu refresh thất bại, session bị xóa và redirect về `/auth`.

6.3. Tính toán tài chính hợp đồng
Nguồn: `be/src/services/contract.service.ts`

```ts
private static calculateFinancials(body: CreateContractBody): ContractFinancials {
  const totalValue = body.quantity * body.pricePerUnit * UNIT_TO_KG[body.unit];
  const commission = (totalValue * CONTRACT_CONFIG.COMMISSION_RATE) / 100;
  const depositAmount = (totalValue * body.depositPercentage) / 100;

  return { totalValue, commission, depositAmount };
}
```

Giải thích
- `totalValue` được tính theo `quantity * pricePerUnit *` hệ số quy đổi đơn vị.
- `commission` hiện đang cố định 3%.
- `depositAmount` phụ thuộc `depositPercentage` của hợp đồng.
- Đây là trung tâm của dòng nghiệp vụ `contract -> escrow -> wallet`.

KẾT LUẬN
========================================================================

PreOnic là một project có độ phức tạp nghiệp vụ khá cao, không chỉ là website CRUD đơn thuần. Hệ thống đã có đầy đủ các khối chức năng có giá trị thực tế: auth, phân quyền, marketplace sản phẩm, hợp đồng, escrow, ví ảo, gateway nạp tiền, messaging, weather alert và partner rating. Về mặt kỹ thuật, dự án đã có nền tảng tốt để phát triển tiếp, và build/backend runtime hiện đang hoạt động được.

Tuy nhiên, để sẵn sàng cho giai đoạn bảo vệ, demo hoặc triển khai nghiêm túc hơn, cần đặc biệt ưu tiên 3 nhóm vấn đề:
- Ổn định session và đồng bộ frontend auth/profile.
- Khóa chính sách phân quyền tranh chấp và hoàn thiện log giao dịch ví.
- Đồng bộ tài liệu/cấu hình/test để giảm rủi ro vận hành.

Nếu cần biến bản này thành:
- báo cáo nộp môn,
- phụ lục API,
- tài liệu Word/PDF có bảng biểu đẹp,
- hoặc version ngắn gọn để thuyết trình,
thì có thể tách chính file TXT này thành nhiều phiên bản sau.
